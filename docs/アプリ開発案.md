# BLEビーコンによる出社自動検知 & Outlook 予定登録システム 計画書

作成日: 2026-05-03

---

## 1. システム概要

### 1.1 目的
オフィス入口に設置した BLE ビーコンを、社員のスマホアプリが受信したことをトリガーに、当該ユーザーの Outlook 当日カレンダーへ「【出社】」という終日予定を自動登録する。打刻・出社実績の見える化、勤怠補助、チーム内の在席共有などに活用する。

### 1.2 システム構成（全体像）

```
[BLEビーコン]  ──BLE電波(iBeacon)──>  [社員スマホアプリ]
 (オフィス入口)                              │
                                            │ HTTPS
                                            ▼
                                    [バックエンドAPI]
                                            │
                                            │ Microsoft Graph API
                                            ▼
                                  [Outlook (M365テナント)]
                                            │
                                            ▼
                            終日予定「【出社】」を登録
```

### 1.3 業務フロー
1. 社員が出社しオフィス入口を通過
2. スマホアプリがビーコンの iBeacon 信号（UUID/Major/Minor）を検知
3. アプリがバックエンド API に「検知通知」を送信
4. バックエンドが Microsoft Graph API 経由でユーザーの Outlook に「【出社】」終日予定を登録（同日重複登録は防止）
5. 完了をアプリへ返却、必要に応じて通知表示

## 2. ハードウェア（BLEビーコン）の選定

### 2.1 選定要件
- **技適マーク必須**（電波法第4条、技術基準適合証明）
- iBeacon フォーマット対応（iOS の `CoreLocation` で省電力に検知できる）
- 設置型・電池駆動（コンセント不要が理想）
- 1台あたり 1〜5 千円程度の入手性
- 電池寿命2年以上

### 2.2 推奨候補

| 候補 | 型番 / 製品 | 価格 | 技適 | 特徴 | 推奨度 |
|---|---|---|---|---|---|
| **本命** | サンワサプライ BLEビーコン **MM-BLEBC3** | 3個セット ¥14,800前後 / 1台あたり約 ¥4,900 | ◯ | 標準タイプ、CR2450コイン電池、小型・薄型（66×30×10.2mm）、iBeacon/Eddystone対応、3個から少量購入可 | ★★★ |
| 対抗 | アプリックス MyBeacon Pro 汎用型 MB004 BT-DR2 | 10台セット ¥39,000（税別）/ 1台あたり ¥3,900 | ◯ | 国内製造、単3乾電池駆動、Bluetooth 5.0、日本語マニュアル。**最小10台から** | ★★ |
| USB給電 | アプリックス MyBeacon USB02W | 1万円前後 | ◯ | USBポート給電、電池交換不要、入口にPC/USB電源があれば候補 | ★★ |

### 2.3 採用案
**サンワサプライ MM-BLEBC3** を第一推奨とする。理由は以下。
- **3個セットから購入可能** で PoC〜小規模導入に最適（アプリックスは10台単位）
- 1台あたり約¥4,900 で本要件にとって十分安価、技適・Bluetooth SIG認証取得済
- 小型・薄型（66×30×10.2mm）で入口への設置が目立たない
- 設定変更はAndroid/iOS用専用アプリから可能（UUID/Major/Minor、電波強度、発信間隔）
- 電池寿命は数ヶ月〜約3年（設定による）。電池切れ通知の仕組みを後述で運用フォロー

### 2.4 機種非依存設計
将来的にビーコン機種を変更しても、アプリ・バックエンドのコード変更を最小化できるよう、以下の方針で実装する。
- **UUID/Major/Minor の判定値はサーバー側DBで管理**（アプリにハードコードしない）
- **RSSI閾値も設定値として外出し**
- アプリは「検知したUUID/Major/Minor/RSSIをサーバーに送る」だけにし、判定ロジックはサーバー側に寄せる
- これにより、本番展開時の電池寿命の長い機種への変更や、別メーカーへの切り替えもサーバー設定変更だけで対応可能

### 2.5 設置設計
- **入口の天井 or 壁面、高さ2〜2.5m** に1〜2台設置（指向性確保のため天井推奨）
- 電波強度は **Tx Power = 中程度（受信半径 約3〜5m）** に絞る
  - 強すぎると通り過ぎただけで検知される、弱すぎると見逃す
- 複数入口がある場合は入口ごとに **Major** を変える（将来の入口別集計に流用可能）
- バックアップとして同じ入口に2台設置（1台が電池切れでも検知継続）
- 残り1台は予備機として保管

## 3. スマホアプリ（開発フレームワーク）の選定

### 3.1 候補比較

| フレームワーク | iBeacon検知の安定性 | バックグラウンド検知 | 開発工数 | 1コードで両OS対応 | M365認証実装の容易さ | 学習コスト | 推奨度 |
|---|---|---|---|---|---|---|---|
| **Expo Dev Client + React Native (TS)** | ○ | ○（要設定） | ◎ 最小 | ◯ | ◯（expo-auth-session） | 低（JS/TS経験者なら即戦力） | ★★★ |
| Bare React Native | ○ | ○ | ◯ | ◯ | ◯ | 中 | ★★ |
| Flutter + flutter_beacon | ○〜◎ | ○ | ◯ | ◯ | △（パッケージ品質まちまち） | 中（Dart習得が必要） | ★★ |
| ネイティブ（Swift+Kotlin） | ◎ 最高 | ◎ OS純正API | ✕ 2OS分必要 | ✕ | ◎（MSAL公式SDK） | 高 | ★ |

### 3.2 採用案: **Expo Dev Client + React Native + TypeScript**

#### 採用理由（他方式との比較）

**vs Bare React Native（Expoを使わない素のRN）:**
- 同じ React Native ベースだが、Expo Dev Client なら **EAS Build で iOS/Android のビルド環境構築が不要**（macOS必須・Xcode設定・Gradle設定などの初期セットアップを省略）
- OTA アップデート（`expo-updates`）でアプリ修正を即時配信でき、社内配布の運用が楽
- ライブラリも `react-native-beacon-radar` のような Expo 互換のものを使えば、ネイティブモジュールを書く必要がない

**vs Flutter:**
- iBeacon検知の安定性は同等レベル
- ただし**Dart の学習コストがある**（社内に Dart 経験者がいない場合、立ち上げに時間がかかる）
- Microsoft Graph API 連携で使う MSAL 互換ライブラリは Flutter より RN/TS のほうが充実
- Web 系開発者がそのまま参加できる Expo の方が**人員アサインの柔軟性が高い**

**vs ネイティブ（Swift + Kotlin）:**
- ネイティブは検知精度こそ最高だが、**iOS/Androidで2OS分のコードを書く必要があり、工数が約2倍**
- 本要件は「ビーコンを受信したら API を叩く」程度のシンプルな処理で、ネイティブの精度差が決定的に効く要件ではない
- 開発工数最優先という方針に最も反する選択

#### Expoの主なメリット（決め手）
1. **開発工数が最小**: 1コードベース、TypeScript、ホットリロード、ビルドはクラウド（EAS Build）任せ
2. **macOS不要でビルド可能**: 開発者のマシン環境を選ばない
3. **OTA配信**: アプリ修正を App Store / Google Play 再申請なしで即時反映可能（社内アプリと相性◎）
4. **iBeacon ライブラリが揃っている**: `react-native-beacon-radar` が Expo Dev Client 互換
5. **M365認証**: `expo-auth-session` で OAuth 2.0 (PKCE) を素直に実装可能
6. **人員確保が容易**: JS/TS 開発者なら即戦力になれる

#### なぜ Expo Go ではダメで、EAS Build が必要なのか
**Expo Go は「お試し用の汎用アプリ」で、限られたAPIしか使えない。**

- **Expo Go**: Expo社が提供する**既存のスマホアプリ**。開発中のJSコードだけを Expo Go アプリ内で動かすため、ネイティブ層に新しい機能を追加できない。BLE/iBeaconのようにOSのネイティブAPIを叩くライブラリは Expo Go には組み込まれておらず、**動かない**
- **Expo Dev Client + EAS Build**: 開発者の**プロジェクト専用のアプリ**を1つビルドする。`react-native-beacon-radar` のようなネイティブライブラリを含めて自分専用のアプリとして配布できる。これにより iBeacon検知が動作する

つまり、Expo Go は「中身を入れ替えできない既製品の箱」、EAS Build は「自分専用に中身を組み立てた箱」というイメージ。本案件は iBeacon という Expo Go では対応できないネイティブAPIを使うため、EAS Build が必須となる。

なお EAS Build はクラウド上でビルドが走るため、開発者のマシンに Xcode や Android Studio をセットアップする必要はなく、Bare React Native と比べて環境構築の手間が大幅に少ない。

#### Expoの注意点（許容できる範囲）
- **iOS Region Monitoring の安定性はネイティブにわずかに劣る** → 入口に複数台ビーコンを設置 + アプリのアクティブ起動時にも補助スキャンすることで実用上問題ないレベルにできる

### 3.3 必要パーミッション
- iOS: `NSLocationAlwaysAndWhenInUseUsageDescription`, `NSBluetoothAlwaysUsageDescription`, Background Modes（`location`, `bluetooth-central`）
- Android: `ACCESS_FINE_LOCATION`, `BLUETOOTH_SCAN`, `BLUETOOTH_CONNECT`, `FOREGROUND_SERVICE`, `POST_NOTIFICATIONS`

### 3.4 主要ライブラリ
- iBeacon検知: `react-native-beacon-radar`
- 認証（OAuth 2.0 PKCE）: `expo-auth-session`
- Secure Storage: `expo-secure-store`
- バックグラウンドタスク: `expo-task-manager` + `expo-location`
- HTTP クライアント: `axios` または `ky`

## 4. バックエンド & Outlook 連携

### 4.1 構成
- **言語/フレームワーク**: Node.js (TypeScript) + Hono（軽量・Lambda最適化）
- **ホスティング**: AWS Lambda + API Gateway
- **DB**: Amazon DynamoDB または Aurora Serverless v2（PostgreSQL互換）
- **認証**: Microsoft Entra ID（旧Azure AD）による Microsoft SSO（OAuth 2.0 認可コードフロー、PKCE 必須）

### 4.2 言語選定: なぜ TypeScript か（Ruby/Python との比較）

| 観点 | TypeScript (Node.js) | Python | Ruby |
|---|---|---|---|
| アプリ（Expo/RN）との型共有 | ◎ 同じ言語で型定義を共有可能 | ✕ 別言語 | ✕ 別言語 |
| Microsoft Graph SDK の品質 | ◎ 公式SDK が最も成熟 | ◯ 公式あり（やや古い） | △ コミュニティ製のみ |
| AWS Lambda コールドスタート | ◎ 速い（数百ms） | ◯ 中（数百ms〜1秒） | △ 遅め |
| 開発工数 | ◎ アプリと一気通貫 | ◯ | ◯ |
| 推奨度 | ★★★ | ★★ | ★ |

**選定理由**:
1. **アプリと言語が同じ**: Expo/RN 側のTypeScript と一気通貫で開発でき、リクエスト/レスポンスの型定義を `shared/types.ts` のような共通パッケージで一元管理できる。型ズレによるバグを防げる
2. **Microsoft Graph 公式SDK が充実**: `@microsoft/microsoft-graph-client` は TypeScript 向けに最も成熟しており、型補完・非同期処理・エラーハンドリングが整っている
3. **AWS Lambda との相性**: Node.jsは Lambda のコールドスタートが速く、API Gatewayとの組み合わせで標準的な構成。Pythonより速く、Rubyより圧倒的に軽量
4. **小規模APIに最適**: 本案件は「検知通知を受けてGraph APIを叩く」程度の薄いAPI。重い計算処理がないのでNode.jsで十分

### 4.3 認証: Microsoft SSO とは
社員が普段使っている **Microsoft 365 アカウント（Entra ID）でログイン**する仕組み（シングルサインオン）。
- アプリ起動時に「Microsoftでログイン」ボタンをタップ
- Microsoft のログイン画面が開き、メアド・パスワード（必要ならMFA）を入力
- 認証成功後、アプリにアクセストークンが返却される
- そのトークンを使って本人のOutlookカレンダーを操作する権限を得る
- アプリ側はパスワードを一切持たない、Microsoft側でMFA等のセキュリティポリシーが適用される

### 4.4 Microsoft Graph API
- 利用 API: `POST /me/events` または `POST /users/{id}/calendar/events`
- 必要スコープ: `Calendars.ReadWrite`（委任権限）、`offline_access`（リフレッシュトークン取得）
- 終日予定のリクエスト例:

```json
POST https://graph.microsoft.com/v1.0/me/events
Authorization: Bearer {access_token}

{
  "subject": "【出社】",
  "isAllDay": true,
  "start": { "dateTime": "2026-05-03T00:00:00", "timeZone": "Tokyo Standard Time" },
  "end":   { "dateTime": "2026-05-04T00:00:00", "timeZone": "Tokyo Standard Time" },
  "showAs": "free",
  "categories": ["出社"]
}
```

ポイント: `isAllDay: true` の場合、`start`/`end` は **00:00 で同タイムゾーン**、終了は翌日 00:00 にする。`showAs: "free"` で他予定との競合を避ける。

### 4.5 重複登録防止
- 同日に既に「【出社】」イベントがあれば登録スキップ
  - クエリ: `GET /me/events?$filter=subject eq '【出社】' and start/dateTime ge '2026-05-03T00:00:00'`
- バックエンド DB にも `(user_id, date)` でユニーク制約を入れる

### 4.6 段階的認証実装（仮認証 → SSO 移行戦略）
Microsoft SSO の本格実装には、Entra ID 側の設定（アプリ登録、リダイレクトURI設定、スコープの管理者同意取得）と、アプリ側の OAuth 2.0 PKCE フロー実装で**1〜2週間程度**かかる。これを後回しにして、先にアプリの「使い物になるか」を判断するため、**開発初期は仮認証で動かし、後からSSOに差し替える**戦略を採用する。

#### Phase 1: 仮認証フェーズ（PoC〜社内βまで）
- **メールアドレス入力だけで擬似ログイン**（パスワードなし）
- バックエンドで「開発モード」のJWTを発行して保持
- Outlook連携部分は**モック化**（実際には予定登録せず、ログだけ残す）
- 環境変数 `ENVIRONMENT=development` のときだけ有効化、本番ビルドでは無効化
- 目的: 検知ロジック・UI・運用フローの確認

#### Phase 2: 本人Outlook連携フェーズ
- 仮認証 + 開発者本人のMicrosoftアカウントでGraph API動作確認
- まずは1〜2人で実Outlook連携を試して、終日予定登録が期待通り動くかを検証

#### Phase 3: Microsoft SSO 導入フェーズ
- Entra ID にアプリ登録、`Calendars.ReadWrite` スコープの管理者同意取得
- アプリに `expo-auth-session` で OAuth 2.0 PKCE フロー実装
- 仮認証は本番ビルドから完全除去
- 全社展開へ

このアプローチにより、**「アプリが本当に使えるか」を最小コストで先に判断**でき、仕様変更や中止判断の柔軟性が確保できる。

## 5. セキュリティ・プライバシー

| 項目 | 対策 |
|---|---|
| トークン管理 | アクセストークンはアプリ内 Secure Storage（iOS Keychain / Android Keystore）に保存。バックエンドではリフレッシュトークンを暗号化保存 |
| 通信 | 全て HTTPS / TLS 1.2 以上 |
| 偽装ビーコン対策 | UUID は秘匿管理、`Major+Minor` で社内ビーコンを識別。なりすまし防止オプション付ビーコンの利用も検討 |
| 位置情報の扱い | **出社の打刻のみを行い、退社時刻や位置の常時追跡は一切しない** ことをプライバシーポリシーに明記 |
| 退職者対応 | Entra ID 連携でアカウント無効化と同時にトークン失効 |

## 6. アプリ配布・テスト方法

### 6.1 開発者登録の必要性

| 方法 | iOS | Android | 開発者登録 | 用途 |
|---|---|---|---|---|
| Expo Go | ◯ | ◯ | 不要 | **iBeacon検知が動かないため本案件NG** |
| Expo Dev Client + EAS Build | △（要証明書） | ◯ | iOS要 | 開発中の動作確認 |
| **TestFlight（iOS）** | ◯ | - | **必要**（年¥14,800） | 最大10,000人までテスト配布 |
| **Internal Testing（Google Play）** | - | ◯ | **必要**（$25買い切り） | 最大100人までテスト配布 |
| APK直接配布（Android） | - | ◯ | 不要（署名は必要） | ユーザー側で「提供元不明アプリ」許可が必要 |

**結論**:
- **Androidだけなら**、開発者登録なしでもAPK直接配布で他人にテスト依頼可能
- **iOSは Apple Developer Program（年¥14,800）がほぼ必須**。最低限のテストでも実機にインストールするには証明書が必要
- 本格的に複数人でテストするなら**両方の開発者登録が現実的**

### 6.2 ブラウザでテストできるか
**できない。** 理由:
- iBeaconの検知には**スマホのBluetoothハードウェア**が必須
- ブラウザのWeb Bluetooth APIはiBeaconを扱えず、バックグラウンド動作も不可
- アプリのUI部分（ログイン画面、設定画面など）の確認だけならExpoのWeb版で可能だが、検知の動作確認には実機が必須

### 6.3 推奨テストフロー
1. **開発初期**: Android端末でAPK直接配布（開発者登録不要）→ 検知ロジックの動作確認
2. **社内βテスト**: Apple Developer Program加入 → TestFlight + Google Play Internal Testing で社内10〜30名へ配布
3. **本番**: 社内MDM経由で全社員へ配布、または App Store / Google Play へ公開（社内アプリなら未公開ストア配布も検討）

## 9. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| iOS のバックグラウンド検知が起動しない | 出社時に検知失敗 | iBeacon の Region Monitoring を採用（OSがアプリを起こす）、フォアグラウンドで補助検知 |
| 電池切れ気づかず検知漏れ | 当日打刻されない | **§10の電池監視機能**で早期警告、月次点検も併用 |
| 隣接ビル・他フロアで誤検知 | 出社していないのに登録 | Tx Power を絞る、RSSI 閾値判定、滞在時間（数秒以上）でフィルタ |
| Graph API のスコープ承認拒否 | 予定登録できない | 管理者同意（Admin Consent）を事前取得、運用ガイド整備 |
| ユーザーが Bluetooth/位置情報をオフ | 検知不可 | アプリ起動時に状態チェック、オフなら通知でガイド |
| 複数入口を短時間で通過 | 二重検知 | DB側で `(user_id, date)` ユニーク制約 + クールダウン時間設定 |

## 10. 電池残量監視機能

ビーコンは電池駆動のため、電池切れによる検知漏れを防ぐ仕組みを設ける。

### 10.1 方式A: Eddystone-TLM による早期警告
- MM-BLEBC3はiBeaconに加え **Eddystone-TLM フォーマットでバッテリー電圧を発信可能**
- スマホアプリが受信時にTLM情報も併せて取得 → バックエンドへ送信
- バックエンドで電圧の閾値判定（例: 残量20%相当を下回ったら警告）
- 管理者にSlackまたはメールで通知 → 電池交換準備

### 10.2 方式B: 死活監視（バックアップ）
- バックエンドで「直近24時間に検知ログがないビーコンID」を検出
- 電池切れまたは故障の可能性 → 管理者通知
- 方式Aで取りこぼしても、最終的な検知漏れを防げる

### 10.3 通知先
- Slackチャンネル（推奨）: 管理者用チャンネルにBOTで投稿
- メール: バックアップとして送信
- 通知内容: 「【ビーコン警告】Major=01 / Minor=02 のビーコン電池残量が低下しています。交換をお願いします。」

## 12. 代替案: BLEゲートウェイ方式（参考）

スマホアプリ方式の代わりに、**BLEゲートウェイ方式**を採用することもできる。本計画ではスマホアプリ方式を本命としているが、将来の拡張・代替検討のために構成・比較を整理しておく。

### 12.1 構成イメージ
```
[BLEビーコン]                       [BLEゲートウェイ]                [クラウド]
 ↓                                   ↓                                 ↓
社員が持ち歩く（社員証/キーホルダー）  オフィス入口の天井に設置         バックエンドAPI
                                   コンセント+Wi-Fi接続              ↓
                                                                    Outlook
```

**役割が逆転する**点に注意:
- **スマホアプリ方式**: 「ビーコン=入口」「受信機=社員のスマホ」
- **BLEゲートウェイ方式**: 「ビーコン=社員」「受信機=入口のゲートウェイ」

社員は **ビーコン付きの社員証やキーホルダー** を身につけ、ゲートウェイがその電波を検知すると「誰が来たか」をクラウドに通知する。スマホアプリは不要。

### 12.2 使用する機器（市販品で構成可能）

| 機器 | 役割 | 推奨製品 | 価格目安 |
|---|---|---|---|
| BLEビーコン（携帯用） | 社員ごとに配布、社員証等に装着 | サンワサプライ MM-BLEBC3 等の小型ビーコン | 1台 約¥4,900 |
| BLEゲートウェイ | 入口に設置、BLE電波を受信→Wi-Fiでクラウドへ | **obniz BLE/Wi-Fi Gateway Gen2.0** | 1台 約¥1万〜¥1.5万 |
| クラウドサービス | ゲートウェイの管理、データ受信API | obniz Cloud（ゲートウェイ購入で無期限ライセンス付属） | 別途バックエンドは自社開発 |

**obniz BLE/Wi-Fi Gateway Gen2.0** はコンセントに挿してWi-Fi設定するだけで動作開始でき、JavaScript/Node.jsで動作をプログラム可能。建設・医療・介護分野で導入実績多数。

### 12.3 スマホアプリ方式との比較

| 観点 | スマホアプリ方式（本命） | BLEゲートウェイ方式 |
|---|---|---|
| 社員の負担 | ◯ アプリインストール+ログインが必要 | ◎ ビーコン付き社員証を持つだけ |
| 個人特定の仕組み | ◎ Microsoft SSO で確実 | △ 「社員=ビーコンID」のDB管理が別途必要 |
| 開発工数 | △ アプリ+バックエンド両方 | ◯ ゲートウェイ側はノーコード〜薄く、バックエンドのみ |
| iOS/Android対応 | △ 両OS対応とテストが必要 | ◎ そもそもアプリ不要 |
| Apple/Google開発者登録 | △ 両方必要（合計¥18,000程度） | ◎ 不要 |
| バックグラウンド検知の不安定性 | △ iOSのRegion Monitoringの制約あり | ◎ ゲートウェイは常時稼働で確実 |
| ビーコンの紛失リスク | ◎ オフィス備品なので紛失しにくい | △ 社員が持ち歩くため紛失・電池切れリスクあり |
| 退職者対応 | ◎ Entra ID連携で即無効化 | △ 物理的にビーコン回収が必要 |
| 在宅勤務時の挙動 | ◎ ビーコン圏外なら何も起きない | ◎ 同上 |
| 拡張性 | ◯ アプリにUIを追加できる | △ ビーコンには表示なし、通知UIなし |
| 電源 | ◎ ビーコンは電池式、運用が手軽 | △ ゲートウェイはコンセント必須、入口にコンセントが必要 |
| 設置工事 | ◎ 不要 | △ Wi-Fiの設定とコンセント確保が必要 |

### 12.4 コスト比較（130名規模の場合）

#### スマホアプリ方式
| 項目 | 金額 |
|---|---|
| 入口用ビーコン×3個セット | 約¥14,800 |
| Apple Developer Program | ¥14,800 / 年 |
| Google Play Developer | $25（約¥4,000） |
| アプリ開発工数 | 中（4〜5ヶ月） |
| バックエンド開発工数 | 中 |
| **イニシャル機材費（人数に依存しない）** | **約¥34,000 + 開発工数** |
| 月額ランニング | AWS約¥1,000〜¥3,000 |

→ **人数が増えても機材費はほぼ変わらない**（入口側ビーコン+開発者登録のみ）

#### BLEゲートウェイ方式
| 項目 | 金額 |
|---|---|
| **携帯ビーコン×130名分（社員ごとに配布）** | **約¥217,000**（3個セット ¥14,800 × 44セット = 132個） |
| BLEゲートウェイ×2台（入口+予備） | 約¥20,000〜¥30,000 |
| アプリ開発 | **不要** |
| バックエンド開発工数 | 小〜中（API+DB+Outlook連携のみ） |
| ビーコン配布・管理運用 | 別途人件費（130個分の配布、退職者からの回収、紛失対応、電池交換管理） |
| **イニシャル機材費（人数に比例して増加）** | **約¥247,000〜¥250,000 + 開発工数** |
| 月額ランニング | AWS約¥1,000〜¥3,000、Wi-Fi利用料 |

→ **1人増えるごとに約¥4,900の追加コスト**（130人だと累計約¥217,000）+ 配布・回収運用の負荷

#### 規模別の機材費差（参考）

| 規模 | スマホアプリ方式 | BLEゲートウェイ方式 | 差額 |
|---|---|---|---|
| 10人 | 約¥34,000 | 約¥80,000 | +約¥46,000 |
| 30人 | 約¥34,000 | 約¥130,000 | +約¥96,000 |

スマホアプリ方式の機材費は人数に依存しないが、BLEゲートウェイ方式は人数に比例して機材費が増える。**人数規模が大きいほどスマホアプリ方式が有利になる構造**。

### 12.5 BLEゲートウェイ方式のメリット

1. **アプリ開発が完全に不要** → 開発工数・期間が大幅削減（推定で全体の30〜40%カット）
2. **iOS/Android両対応の悩みが消える** → 開発者登録費用も不要
3. **バックグラウンド検知の不安定性がない** → ゲートウェイは常時稼働で確実
4. **社員の負担が最小** → アプリインストール・ログイン作業が不要、ビーコン付き社員証を持つだけ
5. **アプリ未インストール社員もカバー** → 全社員に確実に行き渡る
6. **電池監視が容易** → ゲートウェイは常時通電なので死活監視が確実、携帯ビーコン側はTLM受信で集中監視

### 12.6 BLEゲートウェイ方式のデメリット

1. **個人特定の仕組みを別途設計する必要** → 「ビーコンID=社員ID」の管理DB、ビーコン配布・回収フロー
2. **携帯ビーコンの管理コスト** → 130名分の配布・紛失・電池切れ・退職者からの回収などの運用負荷が大きい
3. **ビーコン機材費が人数に比例** → 130名規模ではビーコンだけで約¥217,000、機材費がスマホアプリ方式の約7倍
4. **ゲートウェイ機器のコスト** → 1台¥1万〜¥1.5万、複数入口だと費用増
5. **Wi-Fi接続必須** → オフィスのWi-Fi品質に依存、ネットワーク部門との調整が必要
6. **入口にコンセントが必要** → 既存の電源配置によっては配線工事が発生
7. **ユーザーへの通知UIがない** → 「打刻されました」というフィードバックを社員に返す手段が限られる（メール通知などで補完）
8. **拡張性が低い** → 将来の機能追加（在席ステータス変更、メッセージ表示など）には別の仕組みが必要

### 12.7 どちらを選ぶか

**130名という規模を踏まえると、スマホアプリ方式が有利**。理由は機材費の差が大きいこと、配布・回収運用の負荷が大きいことの2点。BLEゲートウェイ方式が機材費・運用面で有利になるのは、概ね20〜30名以下の小規模なケース。

| こんな場合 | おすすめ方式 |
|---|---|
| 社員が全員スマホを業務利用している、UI/通知でリッチな体験を提供したい | **スマホアプリ方式（本命）** |
| **130名規模・大規模展開で機材費を抑えたい** | **スマホアプリ方式（本命）** |
| 開発工数・期間を最小化したい、小規模（〜30名）で確実性を重視 | BLEゲートウェイ方式 |
| 全社員に確実にカバレッジを広げたい（アプリ未インストール社員ゼロ）、社員の負担を最小化したい | BLEゲートウェイ方式（ただし規模次第） |
| 将来の機能拡張（在席表示、社員間連携など）を視野に入れている | **スマホアプリ方式（本命）** |
| BYOD・社員の私物スマホでの運用に不安がある | BLEゲートウェイ方式 |

### 12.8 ハイブリッド戦略
**スマホアプリ方式で開始 → 運用課題が見えたら一部 or 全体をBLEゲートウェイ方式に移行する** という二段戦略も現実的。
- バックエンドのDB設計を「検知ID（ビーコンID or ユーザーID）→ 社員ID」と疎結合にしておけば、ゲートウェイ方式への移行時もバックエンドの大幅改修は不要
- 機種非依存設計（§2.4）の方針はこの拡張余地も含めて有効
- 130名規模ではBLEゲートウェイ全面移行のコストメリットは薄いが、「アプリ導入を拒否する一部社員にだけビーコン配布」のような部分併用は有効
