# Step 6: バックグラウンド検知 + 自動起動 + 通知

`react-native-beacon-radar` の `useBackgroundScanning: true` で **アプリが background のままでも Ranging を続行** させる。
あわせてアプリ再起動時に **設定が揃っていれば自動でスキャン再開**、**打刻成功はローカル通知**で可視化する。

> 仕様書 §11 にあるように、**iOS でアプリを完全終了 (swipe kill) させた後の検知** は本ライブラリだけでは
> 実現できない。Region Monitoring の `didEnterRegion` が Phase 1 で使うライブラリでは未実装のため、
> Phase 1 では **「再起動後は手動でアプリを 1 回開く」** 運用で受け入れ、**Phase 2 で改善**する。
> 受け入れ基準でも「10回中5回以上検知できれば合格」と緩めてある。

## 完了条件

- [ ] アプリ起動時に Slack トークンと Beacon UUID が両方保存済みなら自動でスキャン開始
- [ ] アプリを background にしても (画面ロックでも) ranging が継続する
- [ ] Beacon 検知 → :kinmu: 設定が成功するとローカル通知が出る
- [ ] Android で Foreground Service の常駐通知が出る (アプリが background でも生きている証拠)
- [ ] iOS で「常に許可」が選ばれている

## 6.1 リスナーモデルへのリファクタ

Step 4 では `startBeaconScanning(config, callbacks)` の引数で UI コールバックを渡していた。
これだと **App.tsx と HomeScreen の両方が検知イベントを受け取りたい** ときに困る (片方しか登録できない)。

そこで:

- スキャナの内部に **`Set<BeaconScanListener>` を持つ** 構造に変更
- `addBeaconScanListener(l)` で購読、戻り値の関数で解除
- `startBeaconScanning(config)` は **コールバック引数を取らない**

```ts
// App.tsx
const unsubscribe = addBeaconScanListener({
  onAction: (r) => { if (r.kind === 'set') notifyKinmuSet(r.jstDate); }
});

// HomeScreen
const unsubscribe = addBeaconScanListener({
  onRawBeacons: setLastRaw,
  onMatched: setLastMatched,
  onAction: setLastActionResult,
});
```

Beacon イベントは 1 つの `DeviceEventEmitter.addListener` から内部で fan-out する。

## 6.2 自動起動 (`autoStartScanIfConfigured`)

`src/beacon/autoStart.ts`:

```ts
export async function autoStartScanIfConfigured(): Promise<AutoStartResult> {
  if (getScanState() === 'scanning') return { started: true, reason: 'already-scanning' };
  const [token, cfg] = await Promise.all([loadSlackToken(), loadBeaconConfig()]);
  if (!token) return { started: false, reason: 'no-token' };
  if (!cfg.uuid) return { started: false, reason: 'no-uuid' };
  try {
    await startBeaconScanning(cfg);
    return { started: true };
  } catch (e) { /* permission / error */ }
}
```

App.tsx の `useEffect(() => { void autoStartScanIfConfigured(); }, [])` で起動時に呼ぶ。

> Phase 1 では「設定があれば自動 ON」固定。Phase 2 では設定画面で ON/OFF を切り替えられるようにする想定。

## 6.3 ローカル通知 (`expo-notifications`)

`src/notifications/setup.ts`:

```ts
Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowBanner: true,
    shouldShowList: true,
    shouldPlaySound: false,
    shouldSetBadge: false,
  }),
});

if (Platform.OS === 'android') {
  await Notifications.setNotificationChannelAsync('punch-in', {
    name: '出社打刻',
    importance: Notifications.AndroidImportance.DEFAULT,
  });
}

await Notifications.requestPermissionsAsync();
```

`notifyKinmuSet(jstDate)` / `notifyKinmuError(message)` でローカル通知を発火。
App.tsx の `addBeaconScanListener` で `kind === 'set'` を見て呼ぶ。

### 通知が必要な理由

- アプリが background のまま打刻されると **画面の Alert は出ない**ので、通知が成功確認の唯一の手段
- iOS で「常に許可」を渡してから Background Mode が動いてないだけのケースを切り分けるためにも有効

## 6.4 iOS の挙動と受け入れ基準

| アプリ状態 | Region Monitoring | Ranging | 検知できるか |
|---|---|---|---|
| Foreground | ✅ | ✅ | ✅ |
| Background (画面ロック含む) | ✅ | `allowsBackgroundLocationUpdates=true` で継続 | ✅ (Background Mode `location` 必須) |
| swipe kill 後 | ✅ (OS が region 入退室を監視) | ❌ ライブラリが didEnterRegion を未実装 | ❌ Phase 1 限界 |
| 端末再起動後 | OS が自動で監視を再開する場合あり | ❌ | △ (場合による) |

> Phase 1 での妥協ライン: **アプリを kill しないで使う**。出社時は画面を 1 回開けば自動スキャンが復帰する。
> 受け入れ基準は「10回中5回以上」と緩めてあるので、まずは普通に使えるかを確認。

## 6.5 Android の挙動

Android は **Foreground Service** ベースなので状況が異なる:

- `useForegroundService: true` (Step 4 で設定済) → 常駐通知が出る
- ユーザーが「アプリのバッテリー使用を許可」しないと、OEM カスタム ROM (Xiaomi, Huawei 等) が殺すことがある
- `FOREGROUND_SERVICE_LOCATION` permission が Android 14+ で必要 (Step 1 で投入済)

Android 13+ の `POST_NOTIFICATIONS` 権限は `requestPermissionsAsync()` で取りに行く。

## 6.6 動作確認 (実機)

### シナリオ A — Foreground

1. アプリ起動 → スキャン自動開始 (ホームの「Beacon スキャン状態 = 稼働中」)
2. iBeacon を近づける → 通知 + Slack ステータス更新

### シナリオ B — Background (kill しない)

1. アプリ起動 → スキャン稼働中の状態で、ホームボタンで background へ
2. iBeacon を近づける (50cm 以内)
3. 数秒〜数十秒以内にローカル通知が来る
4. Slack で自分のステータスが `:kinmu:` になっている

### シナリオ C — 翌日 (冪等性)

1. シナリオ A で打刻完了
2. アプリを kill しない、Beacon を持ったまま再度近づける → **通知が来ない** (`already today` で正常)
3. 設定 > デバッグ > 「本日の打刻記録をクリア」 → 再度近づけると通知が来る

### シナリオ D — 端末再起動

1. アプリ起動・打刻成功・iBeacon が近くにある状態
2. 端末を再起動
3. アプリを **手動で 1 回開く** → 自動でスキャンが再開
4. iBeacon を近づける (実際は隣にいるなら一瞬で検知される)

> Phase 1 では「再起動直後の自動検知」は対象外。Phase 2 で BackgroundLocationUpdates / Region Monitoring 周辺を強化する。

## トラブルシュート

| 症状 | 対処 |
|---|---|
| Background で何も起きない | iOS 設定 > プライバシー > 位置情報 > **常に許可** になっているか確認 |
| Background で 10 分後に止まる | iOS は省電力で `pausesLocationUpdatesAutomatically = false` でも長時間は維持できないことがある。Phase 2 で改善 |
| Android の常駐通知が消える | OS のメモリ圧迫で殺された。OEM のバッテリー最適化を「対象外」にする |
| 通知が出ない (iOS) | `Notifications.requestPermissionsAsync()` で許可を取れたか。設定 > 通知 > 出社番長 で確認 |
| `expo run:ios` 後 Background でビルドが落ちる | `app.json` の `UIBackgroundModes` に `location` が無い。Step 1 を再確認 |
