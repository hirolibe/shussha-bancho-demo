# Step 5: 冪等化（同日重複防止 + クールダウン）

> 順序的には Step 4 (iBeacon検知) の前に作る。
> Beacon 検知は **「`performKinmuAction()` を呼ぶだけ」** にしておきたいので、
> アクション側を先に固める。

仕様書 §5.4 / §11 が要求する 2 つの抑止を 1 つの API にまとめる:

1. **同日重複の抑止**: `last_kinmu_set_date (YYYY-MM-DD JST) === today` ならスキップ
2. **クールダウン**: 直前の試行から 60 秒以内ならスキップ（Slack のレート制限・連打対策）

## 完了条件

- [ ] `performKinmuAction()` が ↑ の判定を踏まえた上で `setKinmuStatus` を呼ぶ
- [ ] 1 回成功 → 連続で再呼出してもネットワークに出ない
- [ ] AsyncStorage から `last_kinmu_set_date` を消すと再度成功できる（デバッグ用）
- [ ] JST の日付計算が端末タイムゾーンに依存しない

## 5.1 `src/actions/kinmuAction.ts`

設計の中核。**Beacon 検知も手動ボタンも、すべてこの関数を呼ぶ。**

### 戻り値の型

```ts
export type KinmuActionResult =
  | { kind: 'set'; jstDate: string }
  | { kind: 'skipped_already_today'; jstDate: string }
  | { kind: 'skipped_cooldown'; remainingMs: number }
  | { kind: 'no_token' }
  | { kind: 'error'; error: SlackError | Error };
```

- **エラーを `throw` しない** ことで、Beacon 検知のような「できれば失敗を握りつぶしたい」呼出でも
  分岐が書きやすい。UI 側は `kind` で switch するだけ。

### JST 日付の作り方

```ts
export function getJstDateString(now: Date = new Date()): string {
  const jst = new Date(now.getTime() + 9 * 60 * 60 * 1000);
  const y = jst.getUTCFullYear();
  const m = String(jst.getUTCMonth() + 1).padStart(2, '0');
  const day = String(jst.getUTCDate()).padStart(2, '0');
  return `${y}-${m}-${day}`;
}
```

- `Intl.DateTimeFormat('ja-JP', { timeZone: 'Asia/Tokyo' })` でも書けるが、
  React Native の Hermes は ICU の追加ビルドが要るためフルの `timeZone` 文字列は信頼できない。
- 上記の **+9h して UTC で読み出す** 方式は、端末タイムゾーンに依存せず JST 日付を出せる。

### 処理フロー

```
1. 今日の JST 日付 today を計算
2. AsyncStorage の last_kinmu_set_date が today と一致 → skipped_already_today
3. AsyncStorage の last_kinmu_attempt_epoch_ms が 60s 以内 → skipped_cooldown
4. SecureStore から token をロード。無ければ no_token
5. last_kinmu_attempt_epoch_ms = now を書き込み (失敗してもクールダウンには記録)
6. (任意) doubleCheckRemote=true なら users.profile.get で既に :kinmu: ならスキップ
7. setKinmuStatus(token) を呼ぶ
8. 成功したら last_kinmu_set_date = today を書き込み → set
9. 失敗したら error
```

### `doubleCheckRemote` を Phase 1 で false 推奨にする理由

- 仕様書 §5.4 が「任意」としている二重ガード
- 1 回の打刻で API を 1 → 2 回に増やす（`profile.get` + `profile.set`）
- AsyncStorage の同日チェックがある以上、二重ガードの効果は **アプリ再インストール後の重複防止** くらい
- レート制限は緩い (Tier 3 ≒ 50req/min) ので影響は小さいが、PoC では落とす方を優先

## 5.2 SettingsScreen にデバッグボタン

開発中、同日に何度もテストしたいので、**`last_kinmu_set_date` をクリアできる UI** を入れる。

```tsx
<Button
  title="本日の打刻記録をクリア"
  color="#c00"
  onPress={async () => {
    await clearLastKinmuSetDate();
    Alert.alert('クリア完了', '本日の打刻記録とクールダウンを消しました');
  }}
/>
```

クリアでは `last_kinmu_set_date` と `last_kinmu_attempt_epoch_ms` の **両方** を消す。
クールダウンが残ったままだと「クリアしたのに再打刻できない」状況が起きるため。

## 5.3 HomeScreen を `performKinmuAction` で呼び直し

Step 3 で書いた手動ステータス変更ボタンを `performKinmuAction()` 経由に差し替える:

```tsx
async function handlePunchIn() {
  const result = await performKinmuAction();
  switch (result.kind) {
    case 'set':                  Alert.alert('打刻完了', `本日 (${result.jstDate}) :kinmu: をセット`); break;
    case 'skipped_already_today': Alert.alert('スキップ', `本日 (${result.jstDate}) はすでに打刻済み`); break;
    case 'skipped_cooldown':      Alert.alert('スキップ', `クールダウン中: あと ${Math.ceil(result.remainingMs / 1000)} 秒`); break;
    case 'no_token':              Alert.alert('未設定', 'トークンを保存してください'); break;
    case 'error':                 /* SlackError をそのまま見せる */ break;
  }
}
```

このボタンは **Step 4 の iBeacon 検知ハンドラと同じ関数を呼ぶ**ので、ここで動けば Beacon 検知側も
Slack まで通る確信が持てる。

## 5.4 動作確認

### シナリオ A — 通常打刻

1. 設定画面で打刻記録をクリア
2. ホームの「出社打刻 (冪等)」をタップ → "打刻完了"
3. もう一度タップ → "本日 (...) はすでに打刻済み"

### シナリオ B — クールダウン

1. 設定画面で打刻記録をクリア
2. ホームの「出社打刻 (冪等)」を **トークン無しで** 連打 → 1 回目 `no_token`、以降は `cooldown`
   - `no_token` でも `last_attempt` を踏むよう書いてあるので、クールダウンが効く

### シナリオ C — タイムゾーン

`getJstDateString()` を Node で単発実行できる関係で、ユニットテストを書きたければ vitest 等で
`new Date('2026-05-09T16:00:00Z')` (= JST 翌日 01:00) を渡すと `2026-05-10` が返ることを確認できる。
